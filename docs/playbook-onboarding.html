SentinAI Playbook System

When the L2 network has a problem, SentinAI automatically finds the right fix using Playbooks — pre-written repair recipes that run without human intervention.

2

3-Layer Playbook Resolution

The matcher searches from most specific to most general, stopping at the first match.

①
🤖 AI-Generated Playbooks EVOLVED
Auto-created from past incident patterns. Instance-specific, confidence-scored, reviewed before active use.
②
⛓ Chain-Specific Playbooks HANDCRAFTED
Tailored for each L2 stack — Arbitrum Nitro, Optimism, ZK Stack, Thanos. Know the exact component names and failure modes.
③
🧩 Core Abstract Playbooks UNIVERSAL
Chain-agnostic. Uses semantic roles like "block-producer" or "tx-submitter" that map to the right component on any chain.
✗
No Match → Escalate to Operator
Incident is flagged and the on-call operator is notified for manual review.

4

Execution Flow

Actions run one by one. If a step fails, fallback steps kick in automatically.

①
Match Playbook
3-layer resolution → playbook found or escalate
②
Pass Safety Gates
Kill switch · Circuit breaker · Cooldown · Rate limits
③
Create Execution Record
Unique ID assigned · status = running
④
Run Primary Actions (sequential)
Check safety level → execute → wait → capture result
⑤
Fallback (if primary failed)
Separate backup action sequence — usually more aggressive (e.g. pod restart)
⑥
Write to Operation Ledger
Updates circuit breaker · Outcome feeds pattern mining for playbook evolution
5

Safety Gates

Five checks run before any action. One failure = playbook blocked.

🔴
Kill Switch
Global on/off. Disables all automation instantly.
⚡
Circuit Breaker
3 consecutive failures → blocked for 24 hours.
⏱
Cooldown
Must wait 10 min between runs of the same playbook.
📊
Hourly Limit
Max 10 total executions per hour.
📅
Daily Limit
Max 50 total executions per day.
6

Action Types

Every action has a safety level that controls whether it runs automatically.

🟢
SAFE
Always runs automatically. Read-only or non-disruptive.
  • health_check
  • collect_logs
  • describe_pod
  • check_l1_gas_price
  • escalate_operator
🟡
GUARDED
Runs automatically only when allowGuardedActions is ON. Changes infrastructure.
  • restart_pod
  • scale_up / scale_down
  • zero_downtime_swap
  • refill_eoa
  • switch_l1_rpc
🔴
MANUAL
Never runs automatically. Requires explicit operator action.
  • config_change
  • rollback_deployment
  • force_restart_all

Trigger signature — a normalized anomaly fingerprint: anomalyType | metricName | zScore bucket | value bucket. Any pattern appearing ≥3 times with consistent outcomes becomes a playbook candidate.

Review Status Lifecycle

Auto-generated playbooks go through a confidence-gated review before being trusted in production.

DRAFT
conf < 0.40
→
PEND
0.40–0.70
→
APPR
0.70–0.90
→
TRUST
≥ 0.90
→
ARCH
terminal

Confidence is calculated from three components:

Base
0.40 — starting point
+ Success
up to +0.20 — from success rate
+ Frequency
up to +0.15 — from occurrence count
Trusted
≥ 0.90 — used in Layer 1
Suspended: Any playbook can be temporarily suspended by an operator and later re-approved. Archived is the only terminal state — it cannot be reactivated.
