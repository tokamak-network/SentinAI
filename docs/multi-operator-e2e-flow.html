Key Design Principles
완전한 데이터 격리
각 운영자 인스턴스가 독립적인 Redis, K8s를 운영. 운영자 A의 장애가 운영자 B에 영향 없음.
하위 호환성 우선
모든 API 파라미터는 optional. operatorAddress 미전달 시 OPERATOR_ADDRESS 환경 변수 fallback. 기존 단일 운영자 배포 변경 없음.
점진적 마이그레이션
Phase 1: 신규 파일만 추가 (기존 수정 없음)
Phase 2: 기존 파일에 optional param 추가
Phase 3: 대규모 multi-tenant (Q3+)
온체인 신뢰 앵커
ERC8004 레지스트리가 운영자 목록의 신뢰 소스. 오프체인 중앙화 없음.

operatorKey() 유틸 동작 원리
// src/lib/agent-marketplace/operator-key.ts /** * Appends operator address suffix to a Redis key. * When no operatorAddress provided, returns key unchanged * for backward compatibility with single-operator deployments. */ export function operatorKey( baseKey: string, operatorAddress?: string ): string { if (!operatorAddress) return baseKey; // Normalize: lowercase, strip 0x prefix const normalized = operatorAddress .toLowerCase() .replace(/^0x/, ''); return `${baseKey}:${normalized}`; } // Examples: // operatorKey('sentinai:logs') // → 'sentinai:logs' (unchanged) // // operatorKey('sentinai:logs', '0xABC123') // → 'sentinai:logs:abc123'
하위 호환성 보장
operatorAddress 미전달 → 기존 키 그대로 반환. 기존 단일 운영자 배포는 코드 변경 없이 동일하게 동작.
정규화 규칙
Ethereum 주소는 대소문자 혼용(0xAbc, 0xABC)이 가능. 정규화로 동일 주소가 항상 동일 키를 생성.
OPERATOR_ADDRESS fallback
operatorAddress 미전달이지만 환경 변수가 있는 경우, 호출자가 process.env.OPERATOR_ADDRESS를 명시적으로 전달해야 적용됨 (암묵적 주입 없음).
