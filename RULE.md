# RULE.md

> 이 파일은 모든 세션에서 가장 먼저 읽어야 하는 전역 규칙 디렉토리입니다.
> 원칙: **최소한의 컨텍스트로 최대한의 정확도**를 달성한다.

---

## 0. 이 파일의 역할

이 RULE.md는 **논리적 디렉토리**다. 규칙과 스킬의 위치를 안내하는 라우터 역할만 한다.
절대로 이 파일 자체에 장문의 지침을 넣지 않는다.

---

## 1. 컨텍스트 관리 (Context Control)

### 1-1. 컨텍스트는 최소화한다
- 현재 태스크에 **직접 필요한 정보만** 로드한다.
- 관련 없는 이전 세션 정보, 불필요한 파일, 무관한 스킬은 절대 참조하지 않는다.

### 1-2. 컴팩션(Compaction) 후 복구 절차
컨텍스트가 압축되었을 때 반드시 다음을 수행한다:
1. 이 RULE.md를 다시 읽는다.
2. 현재 태스크 플랜을 다시 읽는다.
3. 작업 중인 관련 파일들을 다시 읽는다.
4. **절대로 "기억"에 의존하여 가정하지 않는다.**

### 1-3. 가정 금지
- 정보가 부족하면 **반드시 확인**한다.
- 추측으로 빈 칸을 채우지 않는다.
- "아마 이럴 것이다"는 금지. 파일을 직접 읽거나 사용자에게 질문한다.

---

## 2. 리서치와 구현의 분리 (Research ≠ Implementation)

### 2-1. 리서치 페이즈
- 구현 방법이 명확하지 않으면, **먼저 리서치만 수행**한다.
- 리서치 결과를 정리하여 사용자에게 보고한다.
- 이 단계에서는 코드를 작성하지 않는다.

### 2-2. 구현 페이즈
- 구현 지시가 구체적일수록 좋다.
- "인증 시스템 만들어줘" ❌
- "JWT + bcrypt-12 해싱, refresh token 7일 만료 로테이션으로 인증 구현해줘" ✅
- 구체적 지시가 없으면 리서치 결과를 기반으로 사용자의 선택을 받은 후 **새로운 컨텍스트**에서 구현한다.

---

## 3. 편향 제어 (Sycophancy Awareness)

### 3-1. 중립적 프롬프트 원칙
- 존재하지 않는 것을 만들어내지 않는다.
- "버그를 찾아라" → 버그가 없어도 억지로 만들 수 있다.
- "코드 로직을 따라가며 발견 사항을 보고하라" → 중립적 결과를 반환한다.

### 3-2. 없으면 없다고 보고
- 요청받은 것이 존재하지 않으면 **정직하게 "없다"**고 보고한다.
- 사용자를 기쁘게 하기 위해 결과를 조작하지 않는다.

---

## 4. 태스크 완료 기준 (Task Completion Contract)

### 4-1. 완료 조건이 명시될 때까지 끝이 아니다
- 스텁(stub) 구현으로 끝내지 않는다.
- 사용자가 정의한 완료 조건(테스트 통과, 스크린샷 검증 등)을 충족해야 태스크가 종료된다.

### 4-2. 테스트 기반 완료 검증
- 테스트가 정의되어 있으면, **모든 테스트가 통과할 때까지** 태스크를 완료로 간주하지 않는다.
- 테스트를 임의로 수정하지 않는다. (사용자가 명시적으로 허용하지 않는 한)

### 4-3. 태스크 계약서 (Task Contract)
태스크 계약서(`{TASK}_CONTRACT.md`)가 존재하면:
1. 계약서의 모든 항목을 충족한다.
2. 테스트를 실행하고 통과시킨다.
3. 필요시 스크린샷으로 검증한다.
4. 모든 조건 충족 전까지 세션을 종료하지 않는다.

---

## 5. 규칙과 스킬의 구조 (Rules & Skills Directory)

### 5-1. 규칙(Rules) = 선호도/제약
- 하지 말아야 할 것, 반드시 해야 할 것을 정의한다.
- 조건부 로딩: 해당 시나리오에서만 읽는다.

```
IF 세션 시작 → READ rules/session-rules.md
IF 코딩 작업 → READ rules/coding-rules.md
IF 테스트 작성 → READ rules/coding-test-rules.md
IF 테스트 실패 → READ rules/coding-test-failing-rules.md
IF 문서 작성 → READ rules/docs-rules.md
IF 블록체인 인프라 → READ rules/blockchain-rules.md
```

### 5-2. 스킬(Skills) = 레시피/절차
- 특정 문제를 해결하는 구체적 절차를 정의한다.
- 시나리오를 만나면 해당 스킬을 읽고 따른다.

```
IF L2 배포 → READ skills/op-stack-deploy.md
IF SentinAI 모니터링 → READ skills/sentinai-ops.md
IF 토큰 이코노미 분석 → READ skills/token-economics.md
```

### 5-3. 정기 정리 (Hygiene)
- 규칙과 스킬이 축적되면 모순이 생긴다.
- 주기적으로 규칙/스킬을 통합·정리하고 모순을 제거한다.
- 사용자에게 업데이트된 선호도를 확인받는다.

---

## 6. 세션 운영 원칙 (Session Management)

### 6-1. 1 세션 = 1 계약
- 장시간 세션은 컨텍스트 오염을 유발한다.
- 하나의 태스크 계약에 대해 하나의 세션을 운영한다.
- 관련 없는 태스크의 컨텍스트가 섞이지 않도록 한다.

### 6-2. 오케스트레이션
- 새로운 작업이 필요하면 새로운 계약서를 생성한다.
- 각 계약서에 대해 독립적인 세션을 할당한다.

---

## 7. 점진적 개선 (Iterative Refinement)

- 처음부터 완벽한 규칙 세트를 만들려 하지 않는다.
- 베어본으로 시작하고, 문제가 발생할 때마다 규칙을 추가한다.
- 에이전트가 잘못된 행동을 하면 → 규칙으로 추가한다.
- 반복되는 절차가 있으면 → 스킬로 문서화한다.

---

## 8. 의존성 최소화 (Minimal Dependencies)

- 외부 하네스, 플러그인, 패키지를 최소한으로 유지한다.
- 프론티어 모델 회사(Anthropic, OpenAI)가 공식적으로 채택한 기능만 신뢰한다.
- "최신 도구"에 대한 FOMO로 불필요한 의존성을 추가하지 않는다.
- CLI 도구를 주기적으로 업데이트하고, 새로운 공식 기능만 확인한다.

---

## 9. 결과에 대한 책임 (Own The Outcome)

- 에이전트의 출력물을 맹목적으로 신뢰하지 않는다.
- 사용자가 최종 검증의 책임을 가진다.
- 에이전트는 이를 인지하고, 불확실한 부분을 명시적으로 표시한다.

---

## 부록: 멀티 에이전트 패턴 (Multi-Agent Patterns)

복잡한 검증이 필요할 때 다음 패턴을 사용할 수 있다:

1. **탐색 에이전트**: 가능한 모든 이슈를 공격적으로 식별 (superset)
2. **반박 에이전트**: 탐색 결과를 검증하고 거짓 양성을 제거 (subset)
3. **심판 에이전트**: 양쪽 결과를 종합하여 최종 판정

이 패턴은 코드 리뷰, 버그 탐색, 보안 감사 등에 활용한다.
