## Day 1 | 병렬 스프린트 A — 구독 + Feature Gate + Core 추상화

> **원칙**: 타입 계약 먼저 확정 → 병렬 구현. `npx tsc --noEmit` 통과가 완료 기준.

### 에이전트 A1 — 구독 티어 + UI 안내

> **티어 강제에 대한 현실**: SentinAI는 self-hosted이며 사용자가 자신의 LLM API 키를 직접 입력한다.
> 따라서 기술적 기능 차단은 불가능하며 불필요하다. 티어는 **계약 기반 honor system**으로 운영하고,
> Feature Gate는 강제(throw)가 아닌 **UI 안내(nudge)**로만 구현한다.

```
// 티어 모델
// General: 소프트웨어는 무료. 혼자 운영.
// Premium: 소프트웨어는 동일. SentinAI 팀이 함께 운영.
//          차이는 코드가 아닌 사람과 맞춤 설정.

General:    self-hosted, 오픈소스, 무제한 사용
            → 기본 플레이북 + 기본 이상 탐지 임계값 (범용)
            → 커뮤니티 지원 (GitHub Issues)
            → 스스로 설치/설정/운영

Premium:    $299/체인/월 (90일 무료 체험)
            → [맞춤 설정] 커스텀 플레이북 작성 (환경 맞춤, SentinAI 팀 제작)
            → [맞춤 설정] 이상 탐지 임계값 튜닝 (노드 실제 패턴 기반, 오탐 감소)
            → [팀 동행]   전용 Slack 채널 (SentinAI 엔지니어 직접 대응)
            → [팀 동행]   인시던트 co-response (크리티컬 알림 시 팀이 함께 참여)
            → [팀 동행]   월 1회 운영 리뷰 (이상 패턴 분석 + 개선 제안)
            → [우선 접근] 신규 프로토콜 우선 지원 (새 L2 체인, 새 CL 클라이언트)

Enterprise: 협의 — 전담 엔지니어, 커스텀 개발, NDA, 다중 체인 할인
```

## Day 3 | 병렬 스프린트 C — AgentOrchestrator + v2 API + 온보딩 마법사

### 에이전트 C1 — AgentOrchestrator (역할 기반 병렬 에이전트)

> **목적: 병목 제거 + 빠른 액션**.
> 현재 agent-loop.ts는 6단계를 직렬로 실행 — AI 분석(~8초)이 끝나야 실행 단계 시작.
> 역할별 독립 에이전트로 분리하면 AI가 느려도 감지·실행 경로가 차단되지 않는다.

```
[변경 전] — 직렬 파이프라인, 하나의 병목이 전체를 막음
agent-loop.ts (30초 단일 사이클):
  Observe(2s) → Detect(1s) → Analyze-AI(8s) → Plan(1s) → Act(1s) → Verify(1s)
                                    ↑
                          AI 응답 기다리는 동안
                          새 메트릭 수집 중단, 실행 불가

[변경 후] — 역할 분리, 독립 실행 주기
CollectorAgent  (5s 주기)  ── 메트릭 수집만. AI·실행과 무관하게 계속 동작.
DetectorAgent   (10s 주기) ── Z-Score 이상 감지. AI 없이 즉각 판단.
                              → 이상 감지 시 ExecutorAgent에 즉시 신호
AnalyzerAgent   (비동기)   ── DetectorAgent 트리거 시 AI 딥 분석 병행 실행.
                              결과는 Plan·리포트에 보강. 실행을 막지 않음.
ExecutorAgent   (즉시)     ── DetectorAgent 신호 수신 즉시 실행.
                              AI 분석 완료를 기다리지 않음.
VerifierAgent   (즉시)     ── 실행 후 post-condition 검증.
```

### Self-Evolving Playbook System 개요 (`proposal-32`)

```
에이전트 루프가 스스로 운영 데이터를 쌓으면서 플레이북을 진화시키는 구조.
수동 생성이 아닌 VerifierAgent 결과 → OperationLedger → PatternMiner → PlaybookEvolver
의 연속 학습 루프.

30일 플랜이 만들어놓는 것:
  ✓ VerifierAgent          → 학습 데이터 생산자
  ✓ Redis 인스턴스 네임스페이스  → OperationLedger 저장소
  ✓ vector-store (Day 2 B4) → 유사 사례 참조
  ✓ AgentEventBus           → 에이전트 간 결과 전파

30일 이후 추가 필요:
  → OperationLedger (실행 결과 영속 저장)
  → PatternMiner (주기적 패턴 분석)
  → PlaybookEvolver (confidence 기반 플레이북 자동 승격)

기대 효과:
  - 경험이 쌓일수록 동일 이상에 빠르게 대응
  - observe-only → full-auto까지 데이터로 증명하며 단계적 자율성 상승
  - Premium: 팀이 직접 작성 → 검토·승인으로 전환 (10배 확장)
```

## 소유권 모델 (proposal-31 Section 3.3)

> self-hosted 우선 설계. 운영자가 데이터, 비밀, 실행 로그의 소유권을 가진다.

| 경계 | 서비스 제공자 부담 | 운영자(사용자) 부담 |
|------|-------------------|-------------------|
| 배포 아티팩트 | Docker 이미지, 업그레이드 도구, 문서/지원 | 호스팅, 저장소, 트래픽 |
| 데이터 소유 | — | 메트릭, 이벤트, 실행 로그, 인증 비밀 |
| 텔레메트리 | opt-in만 (중앙 수집 기본 비활성화) | — |
| 기본 정책 | `observe-only` (write 권한은 명시적 승격) | 정책 승격 결정 |
| 보안 경계 | 라이선스 서명 + 공개키 | 인스턴스별 감사로그 + 접근 제어 |
