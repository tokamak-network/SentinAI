SentinAI x Agent Marketplace

SentinAI가 운영 API와 에이전트 서비스를 마켓플레이스에서 어떻게 판매하는가

이 문서는 먼저 SentinAI가 판매자인 구조를 설명합니다. SentinAI는 운영 서비스를 마켓플레이스에 등록하고, 구매자는 그곳에서 상품을 찾고 결제한 뒤, 실제 실행은 SentinAI의 API와 에이전트에서 일어납니다. 이 과정에서 ERC-8004 registry 등록, 외부 에이전트 discovery, x402 결제 및 정산이 어떻게 연결되는지도 함께 보여줍니다. 마지막에는 참고용으로 SentinAI가 외부 에이전트를 사오는 반대 케이스도 함께 넣었습니다.

주요 관점: SentinAI가 판매자
보조 관점: SentinAI가 외부 전문 에이전트를 구매
포함 요소: ERC-8004, discovery, x402

1. 관계도

누가 서비스를 등록하고, 누가 discovery를 담당하며, 누가 결제와 정산을 맡고, 실제 실행이 어디서 일어나는지 한눈에 보여줍니다.

2. 판매 / 구매 플로우

registry 등록부터 discovery, x402 결제, 서비스 실행, 정산까지 실제 판매 흐름을 단계별로 보여줍니다.

1
서비스 정의 운영팀이 SentinAI 안에서 판매할 운영 API와 에이전트 서비스, 가격, SLA를 정합니다.
2
Registry 등록 SentinAI가 ERC-8004 registry에 `agentURI`를 등록해 외부가 온체인 기준으로 찾을 수 있게 만듭니다.
3
Discovery 공개 `agent.json`과 catalog가 열리고, 외부 에이전트와 구매자가 서비스, 가격, 네트워크, 결제 방식을 읽습니다.
4
x402 결제 구매자는 x402 기준으로 결제 승인을 보내고, 마켓플레이스 또는 facilitator가 이를 검증합니다.
5
권한 부여와 실행 결제가 확인되면 entitlement가 생기고, 구매자는 SentinAI API를 호출해 실제 분석이나 조치를 받습니다.
6
증빙과 정산 실행 기록과 결과 증빙이 다시 마켓플레이스로 돌아가고, 이를 기준으로 평판 업데이트와 수익 정산이 이뤄집니다.

핵심: 이 플로우의 앞단은 ERC-8004 registry + discovery metadata, 가운데는 x402 결제 검증, 뒷단은 entitlement, 실행 증빙, 정산으로 나뉩니다. 즉 판매는 마켓플레이스에서 보이지만, 실행과 운영 책임은 여전히 SentinAI가 집니다.

3. 반대 케이스

보조 시나리오로, SentinAI가 판매자가 아니라 구매자가 되어 registry와 discovery를 통해 외부 전문 에이전트를 사오는 경우입니다.

SentinAI 필요한 전문 기능과 공급자를 찾음
마켓플레이스 registry, catalog, 가격, 평판 정보를 연결
외부 전문 에이전트 agentURI, discovery metadata, 유료 기능 제공
SentinAI → registry / catalog 탐색 → 외부 에이전트 발견 → x402 결제 → 기능 호출 → 결과 반환
보안 위협 인텔리전스, 지갑 위험도
비용 인프라 비용 최적화, 스케줄 제안
규정 정책 점검, 감사 증빙 생성
체인 특화 OP Stack, ZK Stack, Orbit 전문 기능

핵심: 반대 케이스에서도 구조는 비슷합니다. 다만 이번에는 SentinAI가 buyer가 되어 registry와 discovery surface로 외부 공급자를 찾고, x402 결제로 필요한 전문 기능만 사와 붙입니다.
